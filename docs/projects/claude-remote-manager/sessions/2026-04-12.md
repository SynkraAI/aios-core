## O que foi feito

### Diagnóstico (início da sessão)
Bot Telegram (`@ClaudeCode_Fosc_bot`) estava caindo a cada ~30 minutos. Investigação via logs revelou 3 causas raiz independentes:
1. **bash 3.2 crash** — `${TYPE^^}` (bash 4+) na linha 647 do fast-checker.sh crashava com "bad substitution" em toda mensagem de voz/áudio, causando loop infinito de restarts
2. **Watchdog agressivo** — `PASSIVE_FROZEN_THRESHOLD=600` hardcoded matava sessões saudáveis quando Claude ficava pensando >10 min
3. **setMyCommands payload** — Telegram tem limite **não documentado de ~7KB** (não 100 comandos). Descoberto via binary search contra a API live.

### Fixes aplicados
- `${TYPE^^}` → static `case` statement (POSIX-safe)
- Thresholds configuráveis via `config.json` (lidos por jq com default preservado)
- Binary search por byte budget (`MAX_PAYLOAD_BYTES=6500`) no register-telegram-commands.sh
- `LC_ALL=C cut -b1-253` para descriptions (byte-safe vs char-safe)
- Smoke test (`tests/smoke-bash-syntax.sh`) + pre-push hook (`.githooks/pre-push`)
- `read_int_config` helper com pipeline 5-stage: default → tonumber? → null→0 → int32 clamp → floor + regex guard bash
